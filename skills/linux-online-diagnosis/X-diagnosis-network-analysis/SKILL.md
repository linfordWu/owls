# 在线网络系统故障诊断

本技能旨在利用 Agent 自动化在目标故障主机上实时执行 X-diagnosis 工具栈，不依赖繁量的离线日志解析，而是采用“实时探测、微观交互、多维证伪”的在线手段，精准探测网络及其协议栈子系统在内核驱动与业务逻辑层面的故障。

## 技能目录结构与依赖模块

本技能包含在线诊断脚本与深度参考资料，结构如下：

```text
X-diagnosis-network-analysis/
├── SKILL.md                           # 本技能的主文档与流程规范
├── scripts/
│   ├── net_baseline.sh                # Step 0: 在线基线采集与现场快照
│   ├── tcp_state_summary.sh           # TCP 连接状态汇总分类
│   ├── kernel_drop_counters.sh        # 内核各层丢包计数器快照
│   ├── arp_check.sh                   # ARP 邻居表健康与冲突检查
│   └── port_listen_check.sh           # 端口监听状态与全连接队列积压检查
└── references/
    ├── xdiagnosis_reference.md        # X-diagnosis 在线深度诊断工具手册 (xd_*)
    └── fault_patterns.md              # 典型网络故障模式与根因案例库
```

## ⚠️ 强制执行流程

**必须严格按以下顺序执行，禁止跳过或乱序：**

```
Step 0 (在线状态采集) → Step 1 (场景分类) → Step 2 (深入交互探测) → Step 3 (多维根因校验) → Step 4 (输出报告)
```

**执行规则：**
1. **顺序强制**：必须完成当前步骤并验证通过后，才能进入下一步。
2. **“先采集，后操作”**：在进行任何参数调整或深入探测前，必须先运行 `net_baseline.sh` 保留故障现场。
3. **数据校验**：Step 3 必须通过验证矩阵（E1-E3）审查结论可靠性，严禁在未排除系统瓶颈时武断判定网络故障。
4. **不留冗余**：分析结果展示在回复流中即可，**严禁在目标服务器生成独立分析文件残留**。

**每步完成标志：**
- Step 0：成功获取最近一分钟内的网络健康度基线数据。
- Step 1：确定主要故障场景标签，并完成初步根因假设。
- Step 2：输出故障零点 T0、结构化事件序列矩阵与故障传导链。
- Step 3：输出《根因证据校验表》、原生日志证据及置信度定性。
- Step 4：在界面按固定结构输出最终报告（双重链条：时间链+传导链）。

## Step 0：故障现场实录与基线采集

### 宏观基线采集
**目标**：不加重系统负担，只采核心指标，在分析前保留最原始的故障证据。

**执行命令**：
```bash
# 执行基线采集脚本，获取链路、路由、TCP 状态及内核计数器
bash scripts/net_baseline.sh
```

### 微观精细定位
**目标**：在基线基础上，结合故障表现，手动查看关键内核计数器或配置。
- **丢包场景**：`cat /proc/net/snmp` (关注 IP/TCP 层 Drop 计数)。
- **连接场景**：`ss -it` (查看特定连接的 RTT、MSS 和重传计数)。
- **中断场景**：`cat /proc/interrupts` (查看队列中断是否倾斜)。

## Step 1：场景分类与根因假设构建

基于 Step 0 的结果，利用决策树定位故障发生层级，并构建**根因假设矩阵**。

### 1.1 分层诊断决策树

```text
网络异常
├── ping 网关不通          →  【L1/L2】物理链路 & IP 配置诊断
├── ping 网关通，目标不通   →  【L3】路由 & ARP 诊断
├── ping 通，端口不通       →  【L4/L5】防火墙 & 服务监听诊断
├── 端口通，TCP 被 RST      →  【TCP-RST】xd_tcpresetstack 定位
├── 连接卡顿 / 重传频繁     →  【TCP-SKB】xd_tcpskinfo 分析
├── 偶发丢包 / 抖动         →  【KERN-DROP】xd_ntrace 内核丢包点定位
├── ARP 广播异常            →  【L2-ARP】xd_arpstormcheck 监控
└── 虚拟网卡丢包（virtio）  →  【VRING】xd_netvringcheck 队列监控
```

### 1.2 候选根因假设矩阵

确定场景标签后，必须构建初步的验证矩阵：

| 场景标签 | 候选根因假设（需在 Step 2 中验证） |
| :--- | :--- |
| `NET_TCP_ACCEPT_ERR` | ① 系统全连接队列满 (backlog) ② 应用层 accept 响应过慢 ③ 软中断饱和导致 SYN 丢弃 |
| `NET_TCP_RESET` | ① 防火墙主动拦截 ② 端口未监听且未开启 Keepalive ③ 驱动层队列溢出触发 Reset |
| `NET_PKT_DROP` | ① Netfilter/iptables 规则丢弃 ② 路由查找失败 ③ Socket 缓冲区满 |
| `NET_L2_ARPSTORM` | ① 交换机环路 ② 虚拟机 IP 冲突或 MAC 地址漂移 ③ 网卡驱动处理并发 ARP 能力受限 |

**Step 1 完成标志**：✅ 确定主要场景标签；✅ 完成上述假设矩阵的初步映射。

## Step 2：深度交互分析与传导链重建

Agent **必须**根据 Step 1 选定的方向，通过专项工具进行**时序关联分析**，杜绝盲猜。

### 2.1 时序关联与 T0 锚定 (核心框架)

**故障零点（T0）**定义为最早可观测到网络异常的时间戳。

| 优先级 | 来源 | 说明 |
| :--- | :--- | :--- |
| **P1** | 内核/驱动报错 (`dmesg -T`) | 如 `NIC Link is Down`, `nf_conntrack: table full` 等原生报错。 |
| **P2** | X-diagnosis 工具输出 | 如 `xd_ntrace` 捕获到第一个 Drop 包的时间。 |
| **P3** | SNMP 计数器突变 | `/proc/net/snmp` 中对应计数器的异常增长起点。 |

### 2.2 故障传导链重建 (示例)

以 T0 为基准，构建事件序列矩阵：
- **T0 (内核):** `net_ratelimit` 触发，提示 `SYN flooding`。
- **T0+500ms (xd_ntrace):** 发现 `tcp_v4_rcv` 处由于 Listen 队列满导致包被丢弃。
- **T0+2s (业务):** 应用层日志开始报告 `Connection timed out`。

> ⚠️ **精确定位强制要求**：网络诊断必须追踪到具体的**内核函数名、网卡索引、或是防火墙规则行号**。

## Step 3：根因反思与证据双向校验

执行三重交叉质询，确保不被“环境假象”所误导。

### 3.1 交叉质询铁律 (Cross-Examination Rules)
1. **孤证不立**：内核丢包不能仅凭 `xd_ntrace`，必须辅助 `netstat -s` 或 `/proc/net/snmp` 的计数器增长。
2. **逻辑闭环**：传导链必须完整。例如，“网卡中断高”不能直接推导为“业务卡顿”，必须证明中断导致了“软中断调度延迟”或“排队丢包”。
3. **互斥排异**：确认为网络故障前，必须通过 `top`/`free` 排除 CPU/内存爆满导致的投递失败。

### 3.2 根因证据校验表 (Evidence Validation Matrix)

| 校验维度 | 校验标准要求 | 强制证据格式 |
| :--- | :--- | :--- |
| **E1: 资源背锅隔离** | 异常发生时，CPU (%soft/%sys) 是否正常？ | `[✅/❌ 结果]` + `mpstat/top 数据摘要` |
| **E2: 物理同一性** | 报错的网卡/队列是否对应业务受损的路径？ | `[✅/❌ 结果]` + `ip route/ethtool 对应说明` |
| **E3: 协议双向验证** | 如果是握手失败，对侧的响应包确实入站了吗？ | `[✅/❌ 结果]` + `tcpdump/xd_ntrace 入站证据` |

**拦截机制**：若无法完成上述校验，严禁给出决定性结论，必须标注为“高度疑似 (Suspected)”。

## Step 4：输出结构化诊断报告

**报告结构要求如下：**

1. **Executive Summary (故障摘要)**：事发时间、受影响范围、结论定性。
2. **Fault Chains (双重链条)**：
   - **故障时间链 (Fault Time Chain)**：列出带准确时间戳（精确到秒）的关键节点。
   - **故障传播链 (Fault Propagation Chain)**：描绘物理/逻辑演进路径（例：`全连接队列满 -> 溢出丢包 -> TCP 重传 -> 业务超时`）。
3. **Technical Evidences (核心证据部)**：引用具体内核计数器路径、`xd_*` 关键输出及 E1-E3 校验结果。
4. **Actionable Recommendations (处置建议)**：给出内核参数、规则调整或排查建议。

**诊断完备性检查清单（输出前必检）：**
- [ ] 故障时间链中每一个节点是否都有准确时间？
- [ ] 是否包含以路径/计数器为基础的核心证据？
- [ ] 是否排除了系统资源（CPU/内存）负载干扰？
- [ ] **是否仅在界面输出报告，无任何本地文件残留？**
