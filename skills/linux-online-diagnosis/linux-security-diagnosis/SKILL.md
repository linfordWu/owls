# Linux 安全故障定位技能

## 核心原则

1. **层级化分析**：认证 → 权限 → 网络 → 审计/日志 → 防护策略 → 账户行为 → 内核模块
2. **证据驱动**：所有假设必须通过脚本采集的日志、监控或系统状态验证
3. **脚本优先**：用封装脚本替代逐条命令，减少执行往返，一次采集完整上下文
4. **根因穿透**：给出故障链+时间线，排除项要说明原因，结论要有说服力
5. **闭环输出**：标准诊断结论格式，便于运维人员直接使用

---

## 方法论流程（五阶段）

```
阶段1: 信息收集（症状感知）
  ↓ 执行对应采集脚本
阶段2: 问题分层分析（归因假设）
  ↓ 按安全类别建立假设清单
阶段3: 定位验证（证据分析）
  ↓ 日志/状态/时间线交叉验证
阶段4: 根因分析（故障归因）
  ↓ 确认最小故障路径
阶段5: 提出修复方案（闭环输出）
  ↓ 标准诊断结论 + 分级修复建议
```

## 安全场景分类与诊断入口

根据用户描述的症状，选择对应场景，**立即执行对应采集脚本**：

| 场景编号 | 安全类别 | 典型症状关键词 | 采集脚本 |
|---------|---------|-------------|---------|
| S1 | 用户认证/登录安全 | 无法登录、SSH失败、账号锁定、LDAP/SSSD | `collect_auth.sh [username]` |
| S2 | 权限控制 | 权限拒绝、无法访问文件、服务启动失败 | `collect_permission.sh` |
| S3 | 防火墙与网络访问 | 端口不通、连接被拒、异常端口暴露 | `collect_network.sh` |
| S4 | 系统审计/日志安全 | 日志缺失、auditd异常、日志被篡改 | `collect_audit.sh` |
| S5 | 安全策略与防护 | SELinux拒绝、AppArmor异常、防护服务停止 | `collect_selinux.sh` |
| S6 | 账户滥用/异常行为 | 异常登录、暴力破解告警、未授权提权 | `collect_account_abuse.sh` |
| S7 | 内核/系统漏洞 | 内核模块异常、dmesg报错、补丁缺失 | `collect_kernel.sh` |

> **脚本位置**：`scripts/` 目录。执行方式：`bash scripts/<script_name> 2>&1`
> **综合采集**（场景不明确时）：`bash scripts/collect_all.sh 2>&1`

## 诊断分析步骤

### Step 1：执行采集脚本，获取原始数据

根据症状选择脚本，**一次性获取完整上下文**。若场景跨多类别，可组合执行。

### Step 2：建立假设清单

阅读采集输出后，按如下模板为每个异常建立假设：

```
[假设 #N]
- 异常现象：<具体观测到的异常>
- 可能原因：<配置错误 | 服务异常 | 策略失效 | 资源不足 | 漏洞>
- 验证证据：<来自哪个日志/命令输出的哪一行>
- 优先级：高/中/低
```

### Step 3：时间线重建

从采集数据中提取带时间戳的关键事件，按时序排列：

```
时间线（从早到晚）：
  T1 [时间] - 事件描述 - 来源（日志文件/命令）
  T2 [时间] - 事件描述 - 来源
  T3 [时间] - 事件描述 - 来源
  ...
故障链：事件T1 → 导致T2 → 触发T3 → 最终表现
```

### Step 4：交叉验证，排除干扰项

对每个假设逐一验证，明确**为什么排除**或**为什么确认**。

### Step 5：输出标准诊断结论

使用以下模板输出（必须填写所有字段）：

```
═══════════════════════════════════════════════════════
                  安全故障诊断结论
═══════════════════════════════════════════════════════
【故障根因】  <一句话说明根本原因>
【故障组件】  <具体的服务/配置文件/模块名称>
【故障时间】  <首次异常时间戳，来源于日志>
【影响范围】  <影响的用户/服务/系统功能>
【风险等级】  高/中/低 + 说明

【故障链与时间线】
  T1: [时间] 事件A（来源：xxx日志）
  T2: [时间] 事件B，由A触发/导致（来源：xxx）
  T3: [时间] 故障最终表现（来源：xxx）

【关键证据】
  1. [日志/命令输出片段] → 说明含义
  2. [日志/命令输出片段] → 说明含义

【已排除的假设】
  - 假设X：排除原因（具体证据说明为何排除）
  - 假设Y：排除原因

【为什么一定是此根因】
  <2-4句，逻辑推导：证据A + 证据B + 排除其他假设 → 唯一结论>

【修复建议】（分级）
  P1（立即）：<最紧急的修复动作>
  P2（当天）：<重要但非立即的修复项>
  P3（计划）：<长期加固建议>

【验证方法】
  <修复后如何验证问题解决>
═══════════════════════════════════════════════════════
```

## 跨层故障传播提示

安全故障常跨层传播，分析时注意以下关联：

```
认证失败 → 检查：SSSD/LDAP服务状态 + PAM配置 + 账户锁定状态
权限拒绝 → 检查：DAC权限 + SELinux标签 + ACL规则（三者都可导致EACCES）
日志缺失 → 检查：auditd状态 + 磁盘空间 + 写权限（可能是攻击者清理痕迹）
服务无法启动 → 检查：文件权限 + SELinux + capabilities + 依赖服务
```
