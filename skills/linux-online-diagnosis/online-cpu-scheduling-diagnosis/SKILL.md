# CPU 调度在线诊断 Skill

本技能通过收集系统运行状态数据，帮助诊断 CPU 调度相关的故障问题。

> **重要原则**：本 skill 仅进行信息收集和分析诊断，**不执行任何修复命令**，只给出修复建议。所有修复操作需由用户确认后手动执行。

---

## 文件结构

```
online-cpu-scheduling-diagnosis/

└── scripts/
```

## 诊断流程

### 阶段一：信息收集与场景识别

#### 步骤 1：时间确认

根据用户描述计算故障时间窗口，**必须输出绝对时间**：

| 用户描述 | 时间窗口设定 |
|---------|-------------|
| 明确时间点 | `[故障时间 - 5分钟, 故障时间 + 持续时间 + 5分钟]` |
| "刚才/刚刚" | `[当前时间 - 30分钟, 当前时间]` |
| "间歇性/偶尔" | `[当前时间 - 2小时, 当前时间]` |
| 无法确定 | `[当前时间 - 1小时, 当前时间]` |

- **时间格式**：`YYYY-MM-DD HH:MM:SS`，例如 `2024-01-15 14:00:00`
- **持续性**：持续发生（如 RT 进程死循环）还是间歇性（如定时器风暴）
- **影响范围**：单机、特定进程、特定服务

---

#### 步骤 2：执行信息收集脚本

📄 **脚本**：`scripts/collect_basic_info.sh`

**参数说明**：

| 参数 | 含义 | 是否必填 |
|------|------|---------|
| `-S <时间>` | 故障开始时间 (YYYY-MM-DD HH:MM:SS) | 可选 |
| `-E <时间>` | 故障结束时间 (YYYY-MM-DD HH:MM:SS) | 可选，默认为开始时间+1小时 |
| `-p <PID>` | 精确进程 ID | 可选 |
| `-n <名称>` | 模糊进程名 | 可选 |
| `-s <服务>` | systemd 服务名 | 可选 |

**调用示例**：

```bash
# 系统级全量分析
bash collect_basic_info.sh -S '2024-01-15 14:00:00' -E '2024-01-15 15:00:00'

# 指定进程分析
bash collect_basic_info.sh -S '2024-01-15 14:00:00' -p 12345

# 指定服务分析
bash collect_basic_info.sh -S '2024-01-15 14:00:00' -s nginx
```

**输出说明**：

| 输出类型 | 内容 | 说明 |
|---------|------|------|
| 终端直接输出 | 系统基础信息、负载、进程状态、关键配置 | 简单信息直接展示，附带诊断说明 |
| 文件输出 | 完整内核日志、中断统计、cgroup 详情 | 大量数据保存到 `/tmp/cpu_diag_*/` 目录 |

#### 步骤 3：场景识别

根据信息收集结果，综合分析各维度线索，识别可能的故障场景。

> **注意**：实际故障中多个场景可能同时存在，需要综合判断而非机械对照阈值。

**识别思路**：

1. **先看负载与 CPU 的关系**
   - Load 高 + CPU 空闲 → 关注 D 状态进程、Z 状态进程
   - Load 高 + CPU 忙 → 关注是用户态、内核态还是中断消耗

2. **再看 CPU 使用率分布**
   - wa% 高 → 检查 D 状态进程堆栈，确认 I/O 阻塞原因
   - si% 高 → 检查软中断类型，确认是网络、定时器还是 RCU
   - irq% 高 → 检查硬中断来源，确认是网卡、磁盘还是其他
   - us% 高 → 检查哪个进程消耗 CPU
   - sy% 高 → 检查系统调用、内核模块、printk 日志

3. **检查特殊进程状态**
   - D 状态进程 → 查看堆栈确认阻塞原因
   - Z 状态进程 → 查看父进程状态
   - RT 进程 → 检查是否独占 CPU
   - R 状态进程长期存在 → 检查是否死循环

4. **检查 cgroup 限流**
   - throttled_time > 0 → 容器/进程组被限流

**常见场景组合**：

| 现象组合 | 可能场景 | 分析方向 |
|---------|---------|---------|
| Load 高 + CPU 空闲 + D 状态进程多 | I/O 阻塞 | 查看进程堆栈，确认是磁盘 I/O 还是 NFS |
| Load 高 + wa% 高 + D 状态进程多 | 磁盘 I/O 瓶颈 | 检查磁盘健康、I/O 调度 |
| si% 高 + 网络流量高 | 网络软中断过载 | 启用 RPS/RFS、优化网络参数 |
| sy% 高 + printk suppressed | 日志洪泛 | 检查驱动日志级别、控制台配置 |
| RT 进程存在 + 其他进程饥饿 | RT 进程抢占 | 检查 RT 配置和进程行为 |
| throttled_time > 0 + 业务延迟 | cgroup 限流 | 增加 CPU 配额 |

**输出场景识别结果**：

```
识别场景：场景 X - XXXXX
判断依据：
  - 线索1: 观察到的现象
  - 线索2: 观察到的现象
  - 综合判断: 分析结论
```

### 阶段二：分场景深入分析

根据识别的场景，执行对应的深入分析。

---

#### 场景 1：进程长期 D 状态（不可中断睡眠）

**分析步骤**：

1. **查看 D 状态进程列表**（终端输出 Section 6）
   - 关注哪些进程处于 D 状态
   - 记录进程 PID 和命令行

2. **查看进程堆栈**（终端输出 Section 6）
   - 堆栈显示进程在等待什么资源

3. **查看内核日志**（输出目录 `dmesg_full.txt`）
   - 搜索 `blocked for more than`
   - 搜索 `I/O error`、`blk_update_request`
   - 搜索 `NFS server not responding`

**堆栈特征诊断**：

| 堆栈关键字 | 诊断结论 | 修复建议 |
|-----------|---------|---------|
| `io_schedule` + `submit_bio` | 等待磁盘 I/O | 检查磁盘健康、优化 I/O 调度 |
| `nfs_` / `nfs_wait` | NFS 服务器超时 | 检查 NFS 服务器、网络连通性 |
| `mutex_lock` / `rwsem_down` | 内核锁竞争 | 分析锁持有者、优化锁粒度 |
| `journal_` | 文件系统日志提交 | 检查磁盘性能、调整 journal 参数 |

**深入取证**：
```bash
# 实时观察 D 状态进程变化
watch -n 1 "ps -eo pid,ppid,stat,cmd | grep ' D'"

# 查看块设备 I/O 统计
iostat -x 1 5
```

---

#### 场景 2：进程长期 Z 状态（僵尸进程）

**分析步骤**：

1. **查看 Z 状态进程列表**（终端输出 Section 6）
   - 关注僵尸进程数量
   - 记录父进程 PID

2. **查看父进程状态**（终端输出 Section 6）
   - 父进程是否正常运行
   - 父进程是否在 D 状态

**父进程状态诊断**：

| 父进程状态 | 诊断结论 | 修复建议 |
|-----------|---------|---------|
| 父进程在 D 状态 | 父进程阻塞，无法回收 | 先解决父进程阻塞问题 |
| 父进程已退出（PPID=1） | 孤儿僵尸 | 重启 init 或手动清理 |
| 父进程运行正常 | 未调用 wait() | 修复父进程代码添加信号处理 |

**深入取证**：
```bash
# 查看僵尸进程的父进程详情
ps -eo pid,ppid,stat,cmd | awk '$3 ~ /^Z/ {print "Zombie:", $1, "Parent:", $2}'
```

---

#### 场景 3：软中断过载

**分析步骤**：

1. **查看 CPU 使用率**（终端输出 Section 3）
   - 关注 si% 列是否偏高（>10% 需关注）

2. **查看软中断统计**（终端输出 Section 5）
   - 关注哪个软中断类型计数最高
   - 对比各 CPU 核的分布是否均衡

**软中断类型诊断**：

| 软中断类型 | 诊断结论 | 修复建议 |
|-----------|---------|---------|
| `NET_RX` 猛增 | 网络收包过载 | 启用 RPS/RFS、增加网卡队列、调优网络参数 |
| `NET_TX` 猛增 | 网络发包过载 | 优化发包逻辑、检查网络拥塞 |
| `TIMER` 猛增 | 定时器风暴 | 检查应用定时器使用、减少高频定时器 |
| `RCU` 猛增 | RCU 回调积压 | 检查长时间关抢占、增加 RCU 线程 |
| `SCHED` 猛增 | 调度负载高 | 检查进程数量、优化调度参数 |

**深入取证**：
```bash
# 实时监控软中断变化
watch -n 1 "cat /proc/softirqs"

# 查看软中断处理时间分布
cat /proc/softirqs | awk '{for(i=2;i<=NF;i++) sum[i]+=$i} END {for(i=2;i<=NF;i++) print i-1, sum[i]}'
```

---

#### 场景 4：硬中断过载

**分析步骤**：

1. **查看 CPU 使用率**（终端输出 Section 3）
   - 关注 irq% 列是否偏高（>5% 需关注）

2. **查看硬中断统计**（终端输出 Section 5）
   - 关注哪个中断源计数最高
   - 检查中断是否集中在某个 CPU 核

**中断源诊断**：

| 中断源类型 | 诊断结论 | 修复建议 |
|-----------|---------|---------|
| 网卡中断猛增 | 网络流量高 | 启用中断合并、配置 RSS/RPS、多队列网卡 |
| 磁盘中断猛增 | I/O 负载高 | 优化 I/O 模式、使用异步 I/O |
| 串口中断猛增 | 控制台输出过载 | 移除串口控制台、降低 printk 级别 |
| 时钟中断异常 | 高精度定时器过多 | 检查 hrtimer 使用 |

**深入取证**：
```bash
# 实时监控硬中断变化
watch -n 1 "cat /proc/interrupts | head -20"

# 查看中断亲和性配置
for irq in $(ls /proc/irq/); do
    echo "IRQ $irq: $(cat /proc/irq/$irq/smp_affinity 2>/dev/null)"
done
```

#### 场景 5：RT 进程死循环

**分析步骤**：

1. **查看 RT 进程列表**（终端输出 Section 7）
   - 是否存在 RT 进程
   - RT 进程的调度策略（RR/FIFO）
   - RT 进程 CPU 使用率

2. **查看 RT 配置**（终端输出 Section 4）
   - `sched_rt_runtime_us` 配置值
   - -1 表示 RT 无限制（危险）

**RT 配置诊断**：

| 配置状态 | 诊断结论 | 修复建议 |
|---------|---------|---------|
| RT 进程 100% CPU | RT 进程死循环 | 修复代码、添加 sched_yield |
| sched_rt_runtime_us = -1 | RT 无限制，可能饿死其他进程 | 设置合理限制（如 950000） |
| RT 进程优先级过高 | 影响系统服务 | 降低优先级或改用普通调度 |

**深入取证**：
```bash
# 查看 RT 进程调度详情
chrt -p <PID>

# 实时监控 RT 进程
watch -n 1 "ps -eo pid,rtprio,stat,%cpu,cmd | grep -v '-' | head -10"
```

---

#### 场景 6：R 状态死锁/死循环

**分析步骤**：

1. **查看 R 状态进程**（终端输出 Section 6）
   - 哪些进程长期处于 R 状态
   - CPU 使用率分布

2. **查看进程状态统计**（终端输出 Section 6）
   - R 状态进程数量是否异常

**死锁特征诊断**：

| 特征 | 诊断结论 | 修复建议 |
|-----|---------|---------|
| CPU 高 + 自愿切换低 | 进程不主动让出 CPU | 检查是否死循环 |
| 单核 CPU 100% | 单线程死循环 | 优化业务逻辑 |
| 多核 CPU 100% | 多线程竞争或死循环 | 分析锁竞争、优化算法 |

**深入取证**：
```bash
# 观察上下文切换
pidstat -w 1 -p <PID>

# 函数采样分析
perf top -p <PID>

# 详细性能分析
perf record -g -p <PID> -- sleep 5
perf report
```

#### 场景 7：用户态过载

**分析步骤**：

1. **查看 CPU 使用率分布**（终端输出 Section 3）
   - us%（用户态）占比
   - 哪个进程消耗最多 CPU

**过载类型诊断**：

| 特征 | 诊断结论 | 修复建议 |
|-----|---------|---------|
| us% 高 + 单进程 CPU 高 | 计算密集型应用 | 优化算法、增加 CPU 资源 |
| us% 高 + 多进程 CPU 高 | 多进程计算密集 | 增加资源或优化并发 |

**深入取证**：
```bash
# 热点函数分析
perf top

# 系统调用分析
strace -c -p <PID>
```

---

#### 场景 8：内核态过载

**分析步骤**：

1. **查看 CPU 使用率分布**（终端输出 Section 3）
   - sy%（内核态）占比

2. **查看内核日志**（输出目录 `dmesg_full.txt`）
   - 是否有异常日志

**过载类型诊断**：

| 特征 | 诊断结论 | 修复建议 |
|-----|---------|---------|
| sy% 高 + 系统调用频繁 | 系统调用开销大 | 减少系统调用、使用批量操作 |
| sy% 高 + 无明显进程 | 内核处理开销 | 检查内核模块、驱动问题 |

**深入取证**：
```bash
# 热点函数分析
perf top

# 内核函数分析
perf top -k /path/to/vmlinux
```

---

#### 场景 9：cgroup CPU 限流

**分析步骤**：

1. **查看 cgroup CPU 统计**（终端输出 Section 8）
   - throttled_time 是否大于 0
   - CPU 配额（quota）与周期（period）比值

**限流诊断**：

| 指标状态 | 诊断结论 | 修复建议 |
|---------|---------|---------|
| throttled_time > 0 且增长 | 配额不足 | 增加 CPU 配额 |
| quota < period | 配额过小 | 修正配置 |
| nr_throttled 数量大 | 频繁限流 | 调整配额或优化业务 |

**深入取证**：
```bash
# 检查 CPU 配额配置
cat /sys/fs/cgroup/cpu/<path>/cpu.cfs_quota_us
cat /sys/fs/cgroup/cpu/<path>/cpu.cfs_period_us

# 实时监控限流
watch -n 1 "cat /sys/fs/cgroup/cpu/<path>/cpu.stat"
```

---

#### 场景 10：Printk 打印过载

**分析步骤**：

1. **查看 printk 配置**（终端输出 Section 4）
   - printk 日志级别
   - 控制台配置

2. **查看内核日志**（输出目录 `dmesg_full.txt`）
   - 是否有大量重复日志
   - 是否有 `printk: messages suppressed`

**日志洪泛诊断**：

| 特征 | 诊断结论 | 修复建议 |
|-----|---------|---------|
| 特定驱动大量打印 | 驱动日志洪泛 | 调整驱动日志级别 |
| console 级别过高 | 日志输出到控制台 | 降低 console_loglevel |
| 串口控制台 | 串口输出阻塞 | 移除串口控制台 |

**深入取证**：
```bash
# 检查 printk 级别
cat /proc/sys/kernel/printk

# 检查控制台配置
cat /proc/cmdline | grep console

# 统计日志输出速率
dmesg -w | pv -l > /dev/null
```

---

#### 场景 11：Load 异常但 CPU 空闲

**分析步骤**：

1. **查看负载与 CPU 对比**（终端输出 Section 2、3）
   - Load/Cores 比值
   - CPU 空闲率

2. **查看进程状态分布**（终端输出 Section 6）
   - D 状态进程数量
   - Z 状态进程数量

**Load 异常诊断**：

| 特征 | 诊断结论 | 修复建议 |
|-----|---------|---------|
| D 状态进程多 | I/O 阻塞导致 | 检查磁盘、网络 I/O |
| Z 状态进程多 | 僵尸进程堆积 | 清理僵尸进程 |
| Load 高 + CPU 空闲 + 无 D/Z | 可能是统计异常 | 检查内核版本 |
