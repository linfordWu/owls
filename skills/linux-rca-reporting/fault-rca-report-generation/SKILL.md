# 故障诊断根因分析和报告生成 Skill

本技能通过对前置收集到的诊断证据（如日志片段、指标异常、系统状态等）进行关联分析，推断导致系统故障的根本原因，评估业务影响范围，并最终生成标准化的故障诊断报告。

> **重要原则**：本 skill 主要执行逻辑推理与报告生成。所有分析需基于已有的客观证据数据，严禁无中生有或臆测。

## 诊断流程

### 阶段一：证据收集与关联 (Evidence Collection)

#### 步骤 1：输入数据解析

读取当前任务输入的1个或多个执行结果或诊断输出文件。
将零散的命令行输出、日志片段或监控指标归一化为内部证据实体。

#### 步骤 2：时间线重建 (Timeline Reconstruction)

- 提取各证据中包含的时间戳，统一时区。
- 构建包含：首次异常发生、告警触发、故障蔓延、尝试恢复等关键节点的事件时间轴。
- **强制约束**：时间必须格式化为绝对时间，如 `YYYY-MM-DD HH:MM:SS`。

---

### 阶段二：根因推断与影响评估 (Root Cause Inference & Impact)

#### 步骤 1：构建故障传导链路

以“现象驱动”为核心，自底向上或自顶向下构建逻辑因果树：
- **起点**：故障表象（如：支付接口 500 报错）
- **中间链路**：性能瓶颈 / 资源耗尽（如：DB连接池满）
- **根因候选**：引发瓶颈的最底层技术原因（如：某表缺少索引导致全表扫描）

#### 步骤 2：假设验证与置信度评估

针对提取的根因候选，列出支持证据（正向）与反对证据（反向）：
- 若有决定性证据且能解释所有现象：高置信度 (🟢)
- 仅有间接证据，无明显反证：中置信度 (🟡)
- 存在相互矛盾的现象或证据不足：低置信度 (🟠)

#### 步骤 3：业务影响评估

- **影响范围**：受影响的具体微服务、数据库节点或物理机。
- **故障时段**：故障开始时间至结束时间（或当前时间）。
- **严重程度**：根据受损面定级（如 P1 / P2 / P3 / Critical / Major 等）。

---

### 阶段三：输出标准故障诊断报告

在经过上述推理后，必须严格按照以下结构输出最终报告。

[针对每一个**曾被真实排查过**的假设，记录验证过程。此处为示例：]

#### 假设 A：网络层故障（基于当时网络告警日志的推断）

> 🧪 假设：机房网络抖动或 DNS 异常，导致请求无法到达服务

| 检查项 | 操作（基于真实历史记录） | 结论 |
|--------|------|------|
| 网络连通性 | \`ping db-host\` / \`curl payment-api\` | ✅ 正常（延时 < 2ms） |
| DNS 解析 | \`nslookup payment.internal\` | ✅ 正常（解析到预期 IP） |

**❌ 排除**：基于上述验证数据，网络层正常，非网络问题。

---

#### 假设 C：数据库连接池耗尽 ✅ 确认根因

> 🧪 假设：基于应用日志中的 \`ConnectionTimeoutException\`，推测 DB 连接池已满

**Step 1 — 确认连接池状态**
\`\`\`sql
SHOW STATUS LIKE 'Threads_connected';
-- 实际获取结果：100 / 100（已满）
\`\`\`

**Step 3 — 定位问题 SQL**
\`\`\`sql
EXPLAIN SELECT * FROM orders WHERE status = 1;
-- 实际执行计划：type=ALL → 全表扫描，rows=500万 → 慢日志显示单次耗时 >30s
\`\`\`

**✅ 结论：\`orders.status\` 字段缺失索引，导致全表扫描，连接长期占用，最终连接池耗尽。**

> ⚠️ **反思与自检**：排查树必须与 \`3.2\` 节中**真实执行过的**假设驱动排查路径严格对应。
