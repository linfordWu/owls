# 文件系统在线诊断 Skill

本技能通过收集文件系统状态数据，帮助诊断文件系统相关的在线故障问题。

> **重要原则**：本 skill 仅进行信息收集和分析诊断，**不执行任何修复命令**，只给出修复建议。所有修复操作需由用户确认后手动执行。

## 诊断流程

### 阶段一：信息收集与场景识别

#### 步骤 1：时间确认

根据用户描述计算故障时间窗口，**必须输出绝对时间**：

| 用户描述 | 时间窗口设定 |
|---------|-------------|
| 明确时间点 | `[故障时间 - 5分钟, 故障时间 + 持续时间 + 5分钟]` |
| "刚才/刚刚" | `[当前时间 - 30分钟, 当前时间]` |
| "间歇性/偶尔" | `[当前时间 - 2小时, 当前时间]` |
| 无法确定 | `[当前时间 - 1小时, 当前时间]` |

- **时间格式**：`YYYY-MM-DD HH:MM:SS`，例如 `2024-01-15 14:00:00`
- **持续性**：持续发生（如磁盘空间持续满）还是间歇性（如偶发 IO 高峰）
- **影响范围**：特定分区、特定挂载点、特定进程

---

#### 步骤 2：执行信息收集脚本

📄 **脚本**：`scripts/collect_filesystem_info.sh`

**参数说明**：

| 参数 | 含义 | 是否必填 |
|------|------|---------|
| `-S <时间>` | 故障开始时间 (YYYY-MM-DD HH:MM:SS) | 可选 |
| `-E <时间>` | 故障结束时间 (YYYY-MM-DD HH:MM:SS) | 可选，默认为开始时间+1小时 |
| `-m <挂载点>` | 指定挂载点路径 | 可选 |
| `-p <PID>` | 精确进程 ID | 可选 |

**调用示例**：

```bash
# 系统级全量分析
bash collect_filesystem_info.sh -S '2024-01-15 14:00:00' -E '2024-01-15 15:00:00'

# 指定挂载点分析
bash collect_filesystem_info.sh -S '2024-01-15 14:00:00' -m /data

# 指定进程分析
bash collect_filesystem_info.sh -S '2024-01-15 14:00:00' -p 12345
```

**输出说明**：

| 输出类型 | 内容 | 说明 |
|---------|------|------|
| 终端直接输出 | 磁盘空间、inode 使用、IO 性能、关键文件检查、错误日志 | 简单信息直接展示，附带诊断说明 |
| 文件输出 | 完整内核日志、journal 日志 | 大量数据保存到可写目录 |

**日志输出目录自动回退机制**：

脚本会自动检测并选择可写的目录保存日志：

```
1. 优先尝试 /tmp 目录
2. 若 /tmp 不可写，扫描所有挂载点找可写磁盘
3. 若所有磁盘均不可写 → 仅输出到标准输出
```

> **重要**：当所有磁盘均不可写（空间不足/只读/inode耗尽）时，脚本会自动回退到仅输出标准输出模式。此时通过 Ansible 远程执行仍可捕获完整诊断信息。

**Ansible 使用示例**：

```yaml
- name: 收集文件系统故障信息
  shell: bash /path/to/collect_filesystem_info.sh -S "2024-01-15 14:00:00"
  register: fs_diag_result

- name: 保存诊断日志到控制端
  local_action:
    module: copy
    content: "{{ fs_diag_result.stdout }}"
    dest: "./logs/{{ inventory_hostname }}_fs_diag_{{ ansible_date_time.iso8601_basic_short }}.log"
```

#### 步骤 3：场景识别

根据信息收集结果，综合分析各维度线索，识别可能的故障场景。

> **注意**：实际故障中多个场景可能同时存在，需要综合判断而非机械对照阈值。

**识别思路**：

1. **先看错误日志**
   - `No space left on device` / `ENOSPC` → 关注磁盘空间满
   - `inode.*full` → 关注 inode 耗尽
   - `I/O error` / `Buffer I/O error` → 关注磁盘故障或 IO 问题
   - `Segmentation fault` / `cannot open shared object` → 关注关键文件损坏
   - `remounting read-only` → 关注文件系统保护机制触发

2. **再看资源使用情况**
   - 磁盘使用率 ≥ 95% → 空间即将耗尽
   - inode 使用率 ≥ 95% → inode 即将耗尽
   - %util ≥ 90% → IO 瓶颈
   - D 状态进程多 → IO 等待严重

3. **检查关键文件状态**
   - 核心配置文件权限是否正确
   - 动态库是否存在且可执行
   - 文件系统是否被只读挂载

**常见场景组合**：

| 现象组合 | 可能场景 | 分析方向 |
|---------|---------|---------|
| 写操作失败 + 空间使用率≥95% | 磁盘分区空间满 | 清理空间、扩容分区 |
| 无法创建文件 + inode使用率≥95% | inode 耗尽 | 清理小文件、调整 inode 比例 |
| 进程 D 状态 + %util 高 + await 高 | IO 时延高 | 优化 IO、检查磁盘健康 |
| 程序崩溃 + 动态库异常 | 关键文件损坏 | 恢复文件、检查权限 |

**输出场景识别结果**：

```
识别场景：场景 X - XXXXX
判断依据：
  - 线索1: 观察到的现象
  - 线索2: 观察到的现象
  - 综合判断: 分析结论
```

### 阶段二：分场景深入分析

根据识别的场景，执行对应的深入分析。

---

#### 场景 1：磁盘分区空间满（非根分区）

**分析步骤**：

1. **查看磁盘空间使用**（终端输出 Section 1）
   - 哪些分区使用率超过阈值
   - 非根分区是否受影响

2. **分析空间占用**（终端输出 Section 1）
   - 哪些目录占用空间最大
   - 是否有大文件可清理

3. **查看内核日志**（输出目录 `dmesg_full.txt`）
   - 搜索 `No space left on device`
   - 搜索 `ENOSPC`

**空间诊断**：

| 使用率 | 诊断结论 | 修复建议 |
|-------|---------|---------|
| ≥ 95% | 空间即将耗尽 | 立即清理或扩容 |
| 85-95% | 空间紧张 | 计划清理或扩容 |
| 70-85% | 需关注 | 监控并规划 |

**深入取证**：

```bash
# 查看大文件（>100M）
find /data -type f -size +100M -exec ls -lh {} \; 2>/dev/null | sort -k5 -rh | head -20

# 查看最近修改的大文件
find /data -type f -mtime -1 -size +10M -exec ls -lh {} \; 2>/dev/null

# 查看目录空间占用
du -h --max-depth=2 /data | sort -rh | head -20

# 查看被删除但仍占用空间的文件
lsof | grep deleted
```

#### 场景 2：磁盘分区 inode 耗尽

**分析步骤**：

1. **查看 inode 使用情况**（终端输出 Section 2）
   - 哪些分区 inode 使用率超过阈值
   - 是否有分区空间充足但 inode 耗尽

2. **分析小文件分布**（终端输出 Section 2）
   - 哪些目录包含大量小文件
   - 是否有缓存或临时文件堆积

3. **查看内核日志**（输出目录 `dmesg_full.txt`）
   - 搜索 `inode.*full`
   - 搜索创建文件失败的错误

**inode 诊断**：

| 使用率 | 诊断结论 | 修复建议 |
|-------|---------|---------|
| ≥ 95% | inode 即将耗尽 | 清理小文件或重新格式化 |
| 85-95% | inode 紧张 | 计划清理 |
| 70-85% | 需关注 | 监控并规划 |

**深入取证**：

```bash
# 统计各目录文件数量
for dir in /data/*; do
    echo "$dir: $(find "$dir" -type f 2>/dev/null | wc -l) files"
done | sort -t: -k2 -rn | head -10

# 查找小文件（<1KB）
find /data -type f -size -1k 2>/dev/null | wc -l

# 查找空文件
find /data -type f -empty 2>/dev/null | wc -l

# 查看文件系统 inode 信息
tune2fs -l /dev/sdXN | grep -i inode
```

---

#### 场景 3：系统 IO 时延高

**分析步骤**：

1. **查看 IO 统计**（终端输出 Section 3）
   - `%util` 是否持续高位
   - `await` 是否异常高（>20ms）
   - `svctm` 服务时间是否正常

2. **查看 D 状态进程**（终端输出 Section 3）
   - 哪些进程处于不可中断睡眠
   - 进程在等待什么 IO 操作

3. **查看磁盘队列和调度器**（终端输出 Section 3）
   - 队列深度是否合理
   - IO 调度器是否适合当前负载

4. **查看内核日志**（输出目录 `dmesg_full.txt`）
   - 搜索 `I/O error`
   - 搜索 `timeout`

**IO 性能诊断**：

| 指标 | 正常值 | 异常值 | 诊断结论 |
|-----|-------|-------|---------|
| %util | <70% | >90% | IO 瓶颈 |
| await | <10ms | >50ms | IO 响应慢 |
| svctm | <await | >await | 磁盘性能问题 |
| D 状态进程 | 0-1个 | 多个持续存在 | IO 等待严重 |

**深入取证**：

```bash
# 实时监控 IO
iostat -x 1

# 查看进程 IO 排行
iotop -o

# 查看块设备统计
cat /proc/diskstats

# 查看特定进程的 IO
pidstat -d -p <PID> 1

# 检查磁盘健康状态
smartctl -a /dev/sdX

# 查看当前 IO 调度器
cat /sys/block/sdX/queue/scheduler
```

---

#### 场景 4：关键文件配置/库异常

**分析步骤**：

1. **检查关键配置文件**（终端输出 Section 4）
   - `/etc/passwd`、`/etc/shadow` 权限是否正确
   - `/etc/fstab` 配置是否正确
   - 文件是否存在且大小正常

2. **检查核心动态库**（终端输出 Section 4）
   - `libc.so.6`、`ld-linux-x86-64.so.2` 等是否存在
   - 权限是否正确（应为 755 或 555）
   - 文件类型是否正确

3. **测试二进制执行**（终端输出 Section 4）
   - 基本命令是否能正常执行
   - 是否有动态链接错误

4. **检查挂载状态**（终端输出 Section 4）
   - 是否有分区被只读挂载
   - 挂载选项是否正确

5. **查看内核日志**（输出目录 `dmesg_full.txt`）
   - 搜索 `Segmentation fault`
   - 搜索 `cannot open shared object`
   - 搜索 `remounting read-only`

**关键文件诊断**：

| 文件类型 | 正常状态 | 异常状态 | 诊断结论 |
|---------|---------|---------|---------|
| /etc/passwd | 权限 644 | 权限异常 | 权限错误 |
| /etc/shadow | 权限 000/640 | 权限异常 | 安全风险 |
| libc.so.6 | 存在且可执行 | 不存在或不可执行 | 系统不可用 |
| 二进制文件 | 可执行 | 执行失败 | 库依赖问题 |

**深入取证**：

```bash
# 检查文件完整性
rpm -Va  # RPM 系系统
debsums -c  # Debian 系系统

# 检查动态链接
ldd /bin/ls

# 检查库缓存
ldconfig -p

# 检查文件系统一致性
fsck -n /dev/sdXN  # 只读检查

# 查看文件属性
lsattr /etc/passwd

# 检查 SELinux 上下文
ls -Z /etc/passwd
```

## 故障模式速查表

| 故障模式 | 主要表现 | 关键指标 | 诊断命令 |
|---------|---------|---------|---------|
| 磁盘分区空间满 | 业务写操作失败 | 使用率 ≥ 95% | `df -h` |
| inode 耗尽 | 无法创建新文件 | inode 使用率 ≥ 95% | `df -i` |
| IO 时延高 | 进程 D 状态等待 | %util ≥ 90%, await > 50ms | `iostat -x` |
| 关键文件异常 | 服务异常/系统错误 | 文件不存在/权限错误 | `ls -la`, `ldd` |

---

## 注意事项

1. **数据安全**：所有诊断操作均为只读，不会修改系统状态
2. **性能影响**：信息收集脚本会对系统产生轻微 IO 负载
3. **权限要求**：部分命令需要 root 权限
4. **日志完整性**：内核环形缓冲区有大小限制，早期日志可能被覆盖
