## 根因定位 (Root Cause Localization)

本 Skill 负责在上游已给出的故障时间范围和可疑组件基础上，结合日志、调用链 Trace 以及其他诊断证据，识别真正的根因组件、根因发生时间和根因原因。

你的职责聚焦在“根因确认与因果链路解释”阶段，不负责初始可疑组件筛选（由指标/可疑组件识别类 Skill 完成），也不负责最终面向业务方的报告撰写（由根因分析协调器完成）。

当系统已经通过其他能力识别出一个或多个“可疑组件”后，应优先使用本 Skill 对这些候选进行进一步甄别和收敛。

## 目标与职责

- 核心目标：在给定的时间窗口和候选组件集合下，完成从“可疑组件”到“最终根因组件”的收敛过程，包括：
  - 利用时间对齐、调用链关系、日志/Trace 线索等确认根因发生时间。
  - 判断哪些组件只是被影响（受害者），哪些是真正的触发者（根因）。
  - 分析故障从根因组件向下游传播的因果链路。
  - 输出结构化的根因列表，供上游 Orchestrator 和下游评估模块使用。
- **时间对齐（Time Alignment）要求**：
  - **验证数据时间**：在查询日志之前，务必先检查日志文件的时间戳范围（例如读取文件的最后几行时间）。如果上游传入的“当前时间”窗口与日志文件的实际时间不符（例如处理离线历史数据集），**必须**自动调整分析窗口以匹配日志中的最新活动时间。
  - **逆序分析原则**：对于“刚刚/最近”发生的故障，在分析日志时，**必须从日志文件的末尾（最新的时间）开始向前扫描**。
    - 不要从文件开头开始看，以免被旧的报错误导。
    - 确认最后一条日志的时间。如果最后一条日志是 17:35，用户说“刚刚”，则重点分析 17:05-17:35 这一段。
  - 根因发生时间通常早于或等于故障表现时间。在分析日志时，重点关注故障时间点**之前**的 Error/Warning。
  - 务必确保分析的日志时间段与上游确定的故障窗口一致，不要被历史日志（如文件开头遗留的旧日志）误导。

你需要在允许的范围内支持以下场景：
- 单个故障窗口内只有一个根因组件。
- 单个故障窗口内存在多个根因组件（并发或串联故障）。
- 用户问题只关心其中某一维度（例如仅根因组件、仅根因时间、仅根因原因），但你仍可以在内部保持完整的根因结构。

## 输入约定

当本 Skill 被触发时，你通常会从上游协调器或其他模块获得以下信息（全部或部分）：

- 故障时间窗口（开始时间和结束时间）。
- 上游指标/可疑组件识别阶段输出的候选组件列表及其异常时间（如 `fault_start_time`）。
- 用户提供的原始故障描述（包括业务影响、错误现象等）。
- 与本次场景相关的日志、调用链 Trace 或预处理后的摘要信息（如果可用）。
- 场景信息（如数据集类型 OpenRCA、磁盘故障场景、网络异常场景等）。

如果关键输入缺失（尤其是时间窗口或候选组件列表），你应当首先在内部明确自己的假设来源（例如“仅基于用户描述中的时间信息推断”），再继续后续分析。

## 分析要求

在使用本 Skill 进行根因定位时，你应当：

- 充分利用时间对齐：根因发生时间应与用户描述的故障时间、指标异常时间、日志/Trace 中的异常峰值在时间上尽可能一致。
- 利用依赖关系和调用链：优先将故障最早出现且位于上游的组件视为根因候选，避免把下游连锁失败误判为根因。
- 区分“根因事件”和“次生事件”：对于明显由上游问题触发的组件，仅将其保留为证据，而不是根因本身。
- 支持多根因：在有充分证据时，可以返回多个根因条目；但不要为了“看起来复杂”而强行制造多个根因。
- 明确根因原因：用简洁的自然语言概括根因原因，例如“network packet loss”“high CPU usage”“JVM Out of Memory (OOM) Heap”等，而不是模糊描述。
- 给出合理的置信度：在证据不充分时，应降低置信度，而不是做出过度肯定的结论。

## 输出格式（务必严格遵守）

本 Skill 的输出供其他模块或协调器使用，应当是一个 JSON 列表，不要输出任何额外自然语言说明。列表中的每个元素表示一个根因条目，包含如下字段：

```json
[
  {
    "component_name": "根因组件名称",
    "root_cause_time": "根因发生时间，ISO 格式，使用场景指定的时区",
    "root_cause_reason": "根因原因的简要描述",
    "cause_chain_explanation": "从根因到观察到的故障现象的因果链路说明",
    "confidence_score": 0.0
  }
]
```

字段含义：

- `component_name`：被判定为根因的组件或服务名（如 apache02、Tomcat01、MG01 等）。
- `root_cause_time`：根因事件发生的时间点，使用 ISO8601 格式并遵守场景时区（若未说明，默认 UTC+8）。
- `root_cause_reason`：对根因原因的简洁归纳，例如“network packet loss”“high disk I/O read usage”“high CPU usage”“JVM Out of Memory (OOM) Heap”等。
- `cause_chain_explanation`：用自然语言描述从该根因组件开始，故障如何沿依赖关系和调用链传播到其他组件或业务表现的过程。
- `confidence_score`：对该根因条目的置信度评分，范围建议在 0.0–1.0 之间，数值越大表示越有把握。

注意：

- 列表可以包含多个根因条目，也可以为空列表（当你认为在当前信息下无法给出可靠根因时，应返回空列表，而不是猜测）。
- 不要返回字符串形式的 JSON，请直接返回真正的 JSON 结构。
- 如果用户问题只要求某一个维度（例如只问“根因组件是什么”），你仍然可以在内部使用完整的 JSON 结构，只是上游模块可以自行选择是否只使用部分字段。

## 约束与行为

- 只基于实际提供的候选组件、时间窗口、日志/Trace 信息和上下文证据进行判断，不要臆造不存在的组件或事件。
- 当数据不足以支持明确结论时，可以：
  - 降低 `confidence_score`；
  - 返回少量高置信度的根因候选；
  - 或直接返回空列表，但需要在 `cause_chain_explanation` 中简要说明不确定性来源（例如“关键日志缺失”“调用链信息不完整”等）。
- 不负责重新执行指标异常检测或可疑组件识别，你应把这些作为上游输入来使用，而不是在本 Skill 内重复实现。
- 不输出面向最终用户的 Markdown 报告，本 Skill 只输出 JSON 结果，由根因分析协调器等模块负责将这些结果编排成用户可读的报告。

## 可用工具与调用方式

**优先使用原则**：请优先使用scripts里面提供的工具进行数据查询和分析。只有当这些脚本执行失败、返回空结果或明确无法满足需求时，才考虑使用其他系统命令（如 `grep` 等）直接操作数据文件。

本 Skill 在根因定位阶段可以使用以下日志与 Trace 工具，所有工具均通过脚本所在目录自动完成数据加载和时区配置。

### 日志分析工具（log_tool.py）

- 脚本路径：`.claude/skills/root-cause-localization/scripts/log_tool.py`
- 基本调用形式（工程根目录执行）：

```bash
python .claude/skills/root-cause-localization/scripts/log_tool.py <command> \
  --start <开始时间> \
  --end   <结束时间> \
  --dataset-path <数据集路径> \
  --timezone <时区> \
  [其他参数]
```

通用参数（**必须显式指定，以确保上下文正确**）：
- `--start` / `--end`：ISO8601 格式时间，例如 `2021-03-04T01:00:00`。
- `--dataset-path`：对应场景的数据集根目录绝对路径。
  - **重要约束**：
    1. 路径必须指向**具体场景目录**（例如 `datasets/DiskFault`）。
    2. **严禁**指向父目录（如 `datasets`）。
    3. **严禁**指向场景内的日期子目录（如 `DiskFault/2024-05-20`）。
- `--timezone`：数据集对应的时区，例如 `Asia/Shanghai` 或 `UTC`。

可用子命令及功能：

- `summary`
  - 功能：统计时间窗口内日志总量、实体数量、错误数量等，给出整体日志活动概览。
  - 参数：
    - `--start`：开始时间。
    - `--end`：结束时间。
    - `--entity`：可选，按实体 ID 过滤。

- `query`
  - 功能：按时间、实体和内容模式查询原始日志条目，返回多条格式化日志。
  - 参数：
    - `--start`：开始时间。
    - `--end`：结束时间。
    - `--entity`：可选，实体 ID。
    - `--pattern`：可选，日志内容匹配模式（正则，大小写不敏感）。
    - `--limit`：返回日志条目数量上限，默认 20。

- `templates`
  - 功能：在时间窗口内使用 Drain3 挖掘日志模板，统计每个模板出现频次，可选返回参数示例。
  - 参数：
    - `--start`：开始时间。
    - `--end`：结束时间。
    - `--entity`：可选，实体 ID。
    - `--top-n`：返回模板数量上限，默认 50。
    - `--min-count`：模板最小出现次数阈值，默认 2。
    - `--config-path`：可选，Drain3 配置文件路径。
    - `--include-params`：是否输出参数示例（布尔开关）。
    - `--model-path`：可选，预训练 Drain3 模型路径。

### Trace 分析工具（trace_tool.py）

- 脚本路径：`.claude/skills/root-cause-localization/scripts/trace_tool.py`
- 基本调用形式（工程根目录执行）：

```bash
python .claude/skills/root-cause-localization/scripts/trace_tool.py <command> \
  --start <开始时间> \
  --end   <结束时间> \
  --dataset-path <数据集路径> \
  --timezone <时区> \
  [其他参数]
```

通用参数（**必须显式指定，以确保上下文正确**）：
- `--start` / `--end`：ISO8601 格式时间。
- `--dataset-path`：对应场景的数据集根目录绝对路径。
  - **重要约束**：
    1. 路径必须指向**具体场景目录**（例如 `datasets/DiskFault`）。
    2. **严禁**指向父目录（如 `datasets`）。
    3. **严禁**指向场景内的日期子目录（如 `DiskFault/2024-05-20`）。
- `--timezone`：数据集对应的时区。

可用子命令及功能：

- `find_slow_spans`
  - 功能：在时间窗口内查找耗时超过阈值的慢 span，用于识别性能热点。
  - 参数：
    - `--start`：开始时间。
    - `--end`：结束时间。
    - `--entity`：可选，实体 ID。
    - `--min-duration-ms`：最小时长阈值（毫秒），默认 1000。
    - `--limit`：返回 span 数量上限，默认 10。

- `analyze_trace_call_tree`
  - 功能：针对单个 trace，输出调用树结构和各节点耗时。
  - 参数：
    - `--trace-id`：Trace ID。
    - `--start`：用于定位该 trace 的开始时间。
    - `--end`：用于定位该 trace 的结束时间。

- `get_dependency_graph`
  - 功能：基于 Trace 推导实体依赖关系，统计调用次数，形成依赖图。
  - 参数：
    - `--start`：开始时间。
    - `--end`：结束时间。
    - `--entity`：可选，仅关注某个实体相关的依赖边。

- `detect_anomalies_zscore`
  - 功能：使用 Z-Score 在 Trace 时延上检测异常，返回异常 span 列表。
  - 参数：
    - `--start`：开始时间。
    - `--end`：结束时间。
    - `--entity`：可选，实体 ID。
    - `--sensitivity`：灵敏度（0.0–1.0），默认 0.8。

- `identify_bottlenecks`
  - 功能：按实体汇总 Trace 时延，识别对整体延迟贡献度最高的瓶颈服务。
  - 参数：
    - `--start`：开始时间。
    - `--end`：结束时间。
    - `--min-impact-percentage`：最小影响占比阈值（百分比），默认 10.0。

- `train_iforest_model`
  - 功能：在指定时间段上训练 Isolation Forest 模型，学习服务调用时延的正常模式。
  - 参数：
    - `--start`：训练数据开始时间。
    - `--end`：训练数据结束时间。
    - `--save-path`：可选，模型保存路径。

- `detect_anomalies_iforest`
  - 功能：使用训练好的 Isolation Forest 模型在新时间窗口内检测 Trace 异常。
  - 参数：
    - `--start`：分析开始时间。
    - `--end`：分析结束时间。
    - `--model-path`：可选，模型加载路径；未提供时使用内存中的模型。
